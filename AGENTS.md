# AGENTS.md

Инструкции для AI-агентов и разработчиков, работающих с Pairboard. Файл действует на весь репозиторий. Если в подпапке появится собственный `AGENTS.md`, его правила дополняют или переопределяют этот документ для соответствующей части проекта.

## 1. Назначение продукта

Pairboard — локально запускаемая live-coding платформа для ученика и наставника.

Основной сценарий:

1. Ученик регистрируется и создаёт рабочее пространство.
2. Внутри пространства он создаёт именованные задачи-доски на TypeScript или JavaScript.
3. Ученик приглашает наставника по одноразовой ссылке без привязки к email.
4. Оба участника открывают одну доску, видят присутствие друг друга и изменения кода в реальном времени.
5. Решение автоматически сохраняется, поэтому к нему можно вернуться позже.
6. Общий Pomodoro отображается всем участникам рабочего пространства.

Сохраняйте этот продуктовый фокус. Не превращайте проект в универсальную IDE, LMS или видеосервис без отдельного запроса.

## 2. Текущий стек

- Монорепозиторий на npm workspaces.
- Node.js 20+.
- Frontend: React 19, TypeScript, Vite, React Router, Ant Design, Monaco Editor.
- Backend: Express 5, TypeScript, Prisma, PostgreSQL.
- Realtime: Socket.IO.
- Auth: JWT Bearer token, пароль хешируется через bcryptjs.
- Локальная инфраструктура: PostgreSQL 16 в Docker Compose.

## 3. Структура репозитория

```text
apps/
  api/
    prisma/
      schema.prisma      # схема PostgreSQL
      seed.ts            # демонстрационные пользователи и задачи
    src/
      access.ts          # проверки доступа к workspace и board
      auth.ts            # JWT и Express middleware
      config.ts          # переменные окружения
      db.ts              # Prisma client
      index.ts           # Express/HTTP entry point
      realtime.ts        # Socket.IO комнаты и события
      routes.ts          # REST API
  web/
    src/
      components/        # общие UI-компоненты
      pages/             # страницы приложения
      api.ts             # HTTP client
      auth.tsx           # клиентский auth context
      socket.ts          # singleton Socket.IO client
      styles.css         # основная визуальная система
      overrides.css      # небольшие поздние CSS-коррекции
      types.ts           # клиентские доменные типы
docker-compose.yml       # локальный PostgreSQL
```

Не создавайте третий workspace или общий пакет без реальной необходимости. Текущие типы frontend/backend частично дублируются; выделение `packages/shared` оправдано только при существенном росте общего контракта.

## 4. Быстрый запуск

```bash
cp .env.example .env
npm install
docker compose up -d postgres
npm run db:generate
npm run db:migrate -- --name init
npm run db:seed
npm run dev
```

- Web: `http://localhost:5173`
- API healthcheck: `http://localhost:4000/api/health`

Демо-аккаунты после seed:

- `student@pairboard.local` / `pairboard123`
- `teacher@pairboard.local` / `pairboard123`

Не добавляйте `.env` или реальные секреты в git. Для новых переменных обновляйте `.env.example` и этот документ, если переменная влияет на запуск.

## 5. Основные команды

```bash

Команды отдельного workspace:

```

Перед завершением изменения обязательно запустите как минимум:

```bash
npm run typecheck
npm run build
```

При изменении Prisma-схемы дополнительно запустите:

```bash
npm run db:generate
DATABASE_URL='postgresql://pairboard:pairboard@localhost:5432/pairboard?schema=public' \
  npx prisma validate --schema apps/api/prisma/schema.prisma
```

Если Docker или PostgreSQL недоступны, явно сообщите, что интеграционная проверка БД не выполнена. Не утверждайте, что миграция проверена, опираясь только на `prisma validate`.

## 6. Правила backend

### REST API

- Все маршруты находятся под `/api`.
- Публичные маршруты: healthcheck, регистрация и вход.
- Остальные маршруты должны проходить через `requireAuth`.
- Входные данные валидируются Zod-схемами.
- Ошибки для пользователя возвращаются в формате `{ "message": "..." }`.
- Не возвращайте `passwordHash`, JWT secret и внутренние исключения Prisma.
- Для асинхронных Express handlers используйте существующий `asyncRoute` или эквивалентную централизованную обработку ошибок.

### Доступ

- Любая операция с workspace должна проверять владельца или membership.
- Любая операция с board должна проверять доступ через родительский workspace.
- Переиспользуйте `workspaceAccess` и `boardAccess` из `apps/api/src/access.ts`.
- Удаление доски и создание приглашения сейчас доступны только владельцу пространства.
- Приглашение не привязано к email: доступ получает первый авторизованный аккаунт, принявший одноразовую ссылку.

Не полагайтесь на скрытую кнопку во frontend как на механизм авторизации. Проверка прав всегда должна быть на API или Socket.IO сервере.

### База данных

- PostgreSQL является единственной поддерживаемой БД.
- Для локальной разработки допустим `prisma db push`.
- Для будущего staging/production используйте миграции и `prisma migrate deploy`.
- Не редактируйте сгенерированный Prisma Client вручную.
- При изменении схемы проверяйте cascade-удаления, уникальные ограничения и индексы.
- Поля дат должны храниться как `DateTime`; форматирование выполняется на клиенте.

Ключевые отношения:

- `Workspace.ownerId` задаёт владельца.
- `WorkspaceMember` задаёт фактический доступ участников.
- Владелец также добавляется в `WorkspaceMember` с ролью `OWNER`.
- `Board.version` увеличивается при realtime-сохранении.
- `Pomodoro` имеет связь один-к-одному с workspace.

### Realtime

Текущие события Socket.IO:

- `workspace:join`
- `workspace:leave`
- `board:join`
- `board:leave`
- `board:selection`
- `board:selection-clear`
- `board:change`
- `board:saved`
- `presence:update`
- `pomodoro:action`
- `pomodoro:update`

Правила:

- Socket авторизуется тем же JWT, который используется REST API.
- Перед присоединением к комнате и обработкой события проверяйте доступ пользователя.
- Workspace-комнаты именуются `workspace:<id>`.
- Board-комнаты именуются `board:<id>`.
- Не доверяйте `userId`, email или role из payload клиента; берите пользователя из авторизованного socket context.
- Ограничивайте размер пользовательского контента.
- Помните, что один пользователь может открыть несколько вкладок; presence должен дедуплицироваться по `user.id`.

Текст редактора сейчас работает по модели server-authoritative last-write-wins. Не описывайте её как CRDT или полноценное конфликт-устойчивое редактирование. Если задача требует корректного одновременного редактирования одной строки, планируйте переход на Yjs/Hocuspocus и отдельную стратегию persistence.

## 7. Правила frontend

### Общие принципы

- Пишите новый код на TypeScript.
- Используйте функциональные React-компоненты и hooks.
- Не дублируйте HTTP-конфигурацию: используйте `src/api.ts`.
- Не создавайте новый Socket.IO client для каждого компонента: используйте singleton из `src/socket.ts`.
- Защищённые страницы должны оставаться внутри маршрута `Protected`.
- При переходе на login сохраняйте исходный URL, особенно для invite-ссылок.
- Ошибки пользовательских действий показывайте через Ant Design `message` или осмысленное состояние страницы.

### Визуальное направление

Интерфейс — тёмная «учебная консоль», а не стандартная административная панель.

- Основной фон почти чёрный.
- Зелёный используется для live-состояний и primary actions.
- Янтарный используется для Pomodoro и фокуса.
- Основной шрифт: Manrope Variable.
- Моноширинный шрифт: JetBrains Mono.
- Сохраняйте плотную рабочую композицию и тонкие технические границы.
- Ant Design используется как база поведения, но должен соответствовать существующим theme tokens и CSS.
- Не вводите случайные фиолетовые градиенты, белые dashboard-карточки или несогласованные скругления.
- Проверяйте desktop и мобильный breakpoint до завершения заметных UI-изменений.

Большие зависимости редактора и TypeScript runtime загружаются лениво вместе с `BoardPage`. Не переносите Monaco или `typescript` в основной initial bundle без веской причины.

### Monaco и запуск кода

- Поддерживаются только `typescript` и `javascript`.
- Пользовательский код запускается только в браузерном Web Worker.
- Лимит выполнения сейчас составляет 2 секунды.
- Вывод собирается через перехваченный `console.log`.
- Не запускайте пользовательский код через `eval`, `Function` или `child_process` внутри основного API.
- Серверный runner в будущем должен быть отдельным sandbox-сервисом без сети, с лимитами CPU, памяти и времени.

## 8. Безопасность

Текущая auth-схема пригодна для локального MVP, но не для публичного production без доработок.

Перед публичным деплоем нужны:

- access/refresh token flow или защищённые HTTP-only cookies;
- rate limiting для login/register/invite;
- восстановление пароля и подтверждение email;
- аудит и возможность отзыва активных приглашений;
- строгий production `JWT_SECRET`;
- TLS и корректный CORS allowlist;
- аудит логирования, чтобы JWT и пользовательский код не попадали в логи;
- Redis adapter или аналог для Socket.IO при нескольких API-инстансах;
- sandbox для исполнения пользовательского кода.

`npm audit` на момент создания файла сообщает о транзитивных предупреждениях в цепочке Monaco Editor → DOMPurify. Не маскируйте audit-ошибку и не используйте нестабильный nightly Monaco без проверки. Обновите Monaco после выхода совместимой исправленной стабильной версии и повторите сборку/editor smoke-test.

## 9. Тестирование изменений

Автоматического test runner пока нет. Для каждой задачи выбирайте проверки пропорционально риску.

Минимум для любого изменения:

1. `npm run typecheck`
2. `npm run build`

Для API/БД:

1. Prisma validate/generate.
2. Запуск PostgreSQL.
3. Проверка healthcheck.
4. Проверка затронутых маршрутов с реальным JWT.
5. Проверка запрета доступа пользователю вне workspace.

Для realtime:

1. Откройте два аккаунта или две независимые browser-сессии.
2. Проверьте join/leave presence.
3. Измените код с обеих сторон.
4. Перезагрузите страницу и убедитесь, что последнее сохранённое содержимое восстановилось.
5. Запустите, поставьте на паузу и сбросьте Pomodoro с разных клиентов.

При запущенных API и PostgreSQL основной набор REST/realtime-сценариев автоматизирован:

```bash
npm run verify:scenarios
```

Скрипт создаёт отдельные тестовые аккаунты и workspace с уникальными именами. Он не очищает эти данные автоматически, чтобы после сбоя состояние можно было исследовать в PostgreSQL.

Для frontend:

1. Проверьте login/register.
2. Проверьте пустые, loading и error-состояния.
3. Проверьте ширину около 1280 px и мобильную ширину до 760 px.
4. Проверьте отсутствие ошибок в browser console.
5. Для Monaco проверьте TS и JS режимы, автосохранение, console output и timeout бесконечного цикла.

## 10. Рабочий процесс агента

1. Сначала прочитайте `README.md`, этот файл и затронутые исходники.
2. Проверьте состояние рабочей директории; не перезаписывайте чужие изменения.
3. Меняйте минимально необходимое количество модулей.
4. При изменении API синхронно обновляйте frontend contract и типы.
5. При изменении переменных окружения обновляйте `.env.example`.
6. При изменении сценария запуска обновляйте `README.md`.
7. Выполните проверки из раздела тестирования.
8. В итоговом сообщении перечислите сделанное, проверки и всё, что не удалось проверить.

## 11. Критерии готовности

Изменение готово, если:

- сохраняет продуктовый сценарий ученика и наставника;
- права проверяются на сервере;
- пользовательские данные не теряются после reload;
- realtime-события не доступны постороннему workspace;
- TypeScript проходит без ошибок;
- production build собирается;
- документация соответствует командам и фактическому поведению;
- известные ограничения названы явно, а не скрыты.
